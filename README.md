
# VMware Workstation k3s Automated Cluster Deployer

Автоматизированный инструмент «в один клик» для развертывания готового к работе Kubernetes-кластера (`k3s`) на базе локальной инфраструктуры VMware Workstation под управлением Windows. 

Проект гармонично объединяет философию Infrastructure as Code (**Terraform**), автоматизацию конфигураций (**Ansible** через **WSL**) и удобную сервисную обертку на **Python/PowerShell** для управления всем жизненным циклом деплоя.

---

##  Основные возможности

* **Полная автоматизация:** Развертывание от пустой папки до готового кластера k3s одной командой через интерактивное меню.
* **Динамическое масштабирование:** Управление количеством воркер-нод прямо через переменные конфигурации.
* **Категорический SSH Hardening:** Автоматическое отключение аутентификации по паролю, блокировка входа для `root`, генерация уникальных хост-ключей и авторизация исключительно по SSH Ed25519 ключам.
* **Умный сетевой менеджмент:** Динамический парсинг выданных DHCP-адресов внутри VMware, автоматическая фиксация их в статику (`/etc/network/interfaces`) и генерация матрицы хостов `/etc/hosts`.
* **Безопасность данных (SecOps):** Раздельные пароли для разных типов нод, изоляция секретов в локальные авто-загружаемые файлы и шифрование чувствительных данных с помощью `Ansible Vault`.
* **Самоочистка:** Автоматический сброс старых SSH-ключей (`known_hosts`) в WSL перед каждым деплоем для исключения конфликтов аутентификации.

## 🛠 Архитектурная специфика проекта

В ходе разработки инфраструктурной логики был решен ряд критических ограничений используемых инструментов:

1. **Ограничения кастомного провайдера Terraform:** Используемый провайдер `elsudano/vmworkstation` имеет лимиты на работу с путями и не умеет автоматически регистрировать клонированные машины в графическом интерфейсе (GUI) VMware. Это решено внедрением кастомных `local-exec` провайдеров, вызывающих утилиту `vmrun.exe`.
2. **Проблема медленного GRUB и ожидания сети:** Из-за долгого старта загрузчика GRUB в эталонном образе Debian стандартные методы инициализации падали по таймауту. В `main.tf` встроен отказоустойчивый цикл PowerShell (`Get-ValidIP`), который непрерывно опрашивает VMware Tools до тех пор, пока виртуальная машина гарантированно не получит IP по DHCP.
3. **Однопоточный режим API VMware (`vmrest`):** Сервис `vmrest` и сам движок VMware Workstation крайне нестабильно обрабатывают конкурентные параллельные запросы на создание машин. Для предотвращения дедлоков и сбоев API деплой жестко ограничен одним потоком. Воркеры создаются строго последовательно (`depends_on` матрица и флаг `-parallelism=1`).
4. **Сетевая изоляция (VMnet8):** Архитектура кластера требует, чтобы все адаптеры находились строго в сегменте VMware Workstation **VMnet8 (NAT)**. В противном случае нарушается связность между хостом сборщика, WSL-контекстом и нодами кластера.
5. **Генерация уникальных SSH-ключей на эталонном образе:** Оригинальный образ Debian по умолчанию не перегенерирует уникальные SSH-ключи хоста при клонировании. Для исправления этой уязвимости в эталонный образ был внедрен кастомный `systemd` юнит, который проверяет наличие ключей при старте и безопасно генерирует их «на лету», если они отсутствуют.

## 📋 Системные требования

* **ОС:** Windows 10/11 Pro/Enterprise с установленным компонентом **WSL2** (в случае его отсутствия скрипт установит его самостоятельно)
* **Дистрибутив WSL:** `Debian` (с установленным внутри пакетом `ansible`) (в случае его отсутствия скрипт установит его самостоятельно).
* **Гипервизор:** VMware Workstation Pro.
* **Утилиты в Windows:** Python 3.11+, Terraform, winget.

## 🚀 Быстрый старт

1. Склонируйте репозиторий в рабочую директорию на Windows:
   ```bash
   git clone [https://github.com/your-username/your-repo-name.git](https://github.com/your-username/your-repo-name.git)
   cd your-repo-name

   ```

2.  Скачайте [эталонный образ](https://drive.google.com/file/d/1Z3WMkIudptpIibJdSSfpT6tFD5T6uqJL/view?usp=sharing) и импортируйте/откройте его в VMware Workstation, чтобы гипервизор добавил его в свой глобальный список.
    
3.  Запустите главный управляющий скрипт из консоли PowerShell от имени администратора:
    
    PowerShell
    
    ```
    .\run.ps1
    
    ```
    
4.  Выберите **Пункт 1 (Запустить подготовку)**. Скрипт сам проверит окружение, создаст конфигурационные файлы и попросит вас ввести пароли для будущих нод и ключ шифрования для Ansible Vault.
    
5.  После успешного шага подготовки согласитесь на деплой (или выберите **Пункт 2** в меню). Начнется автоматический процесс раскатки инфраструктуры.
    

## 🗂 Структура проекта

-   `run.ps1` — Точка входа. Управляет установкой системных зависимостей (через `winget` и `pip`) и предоставляет интерактивный интерфейс.
    
-   `pre-deploy.py` — Скрипт подготовки. Отвечает за автоматическую регистрацию учетных данных в VMware REST API, запуск API в фоне, поиск родительского Debian-образа и криптографическую защиту секретов.
    
-   `deploy.py` — Оркестратор развертывания. Инициализирует и применяет конфигурацию Terraform, парсит и очищает сетевые кэши, передает управление в подсистему конфигурации.
    
-   `main.tf` — Описание инфраструктуры как кода. Содержит логику декларативного описания Master-ноды и динамического пула Worker-нод.
    
-   `k3s_deploy.yml` — Ansible-плейбук. Производит глубокую настройку ОС, фиксирует сеть, закрывает SSH от несанкционированного доступа и собирает кластер k3s (инициализация Master, чтение токена, подключение ведомых нод).
    
-   `state.json` — Файл сохранения состояний. Позволяет pre-deploy скрипту безопасно возобновлять работу в случае прерывания на промежуточных шагах.

-   `passwords.yaml` и  `passwords.yaml` — Файлы для хранения переменных, паролей и путей, пути и переменные генерируются динамически при использовании run.ps1    

## 🔒 Безопасность и хранение секретов

Проект спроектирован с учетом строгих практик управления доступом:

-   Все мастер-пароли генерируются исключительно в оперативной памяти во время выполнения скриптов деплоя и никогда не хранятся на диске в открытом виде.
    
-   Конфигурационный файл `passwords.yaml`, содержащий хэши паролей для системных учетных записей нод кластера, проходит процедуру симметричного шифрования алгоритмом AES-256 через `ansible-vault` прямо внутри WSL до начала выполнения основных задач автоматизации.
